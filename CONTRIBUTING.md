# Contributing

## Branch and PR flow
- Create small focused branches.
- Open PRs with a clear summary, testing notes, and any schema or env changes.
- Keep one logical change per PR whenever possible.

## Before requesting review
- Run the relevant lint, typecheck, and test commands.
- Update docs, fixtures, and env examples when behavior changes.
- Call out any new trusted source domains or data-governance changes.

## Team agreements
- Prefer straightforward code over premature abstractions.
- Keep source attribution visible in product and data contracts.
- Document major architecture choices in `docs/adr`.
